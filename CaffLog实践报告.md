## 摘要

随着咖啡文化在国内的快速普及，咖啡因摄入量管理成为健康领域的新关注点。FDA建议成年人每日咖啡因摄入不超过400毫克，但普通消费者缺乏科学的估算和追踪手段。本项目设计并实现了一款名为"CaffLog"的微信小程序，为用户提供咖啡因摄入记录与管理服务。

核心功能覆盖三大场景：自制咖啡因计算——用户选择咖啡豆品种、输入重量和冲煮方式，系统自动计算该杯咖啡的咖啡因含量；品牌饮品快速记录——内置星巴克、瑞幸、库迪、麦当劳、Manner等8个品牌的饮品数据库，用户通过品牌→品类→饮品→杯型的选择即可快速记录；每日仪表盘与历史趋势——以Canvas环形进度条展示当日摄入量，以柱状图和折线图展示周度和月度趋势。

项目采用微信小程序原生框架开发，配合云开发提供后端数据服务，数据存储采用云数据库与本地存储双写策略。开发过程中逐一解决了Canvas图表刷新同步、品牌选择联动、数据持久化、编译兼容性等关键技术问题，最终交付了一个可完整运行的小程序应用。

**关键词**：咖啡因追踪；健康管理；微信小程序；云开发；Canvas可视化

## 一、项目概述

### 1.1 项目背景与核心痛点

中国咖啡市场在2025年已突破4000亿元规模，人均咖啡消费量持续攀升。咖啡因作为中枢神经兴奋剂，适量摄入可提神醒脑，但过量摄入会导致心悸、焦虑、失眠等健康问题。FDA与EFSA均建议健康成年人单日摄入不超过400毫克。

当前市场上的健康管理类应用普遍聚焦于热量计算、运动记录和睡眠监测，缺乏专注于咖啡因摄入管理的工具。用户面临三大核心痛点：

**痛点一：自制咖啡的咖啡因含量难以估算。** 一杯手冲咖啡的咖啡因含量受咖啡豆品种（阿拉比卡约1.2%、罗布斯塔约2.2%）、冲煮方式、研磨度、水温等多种因素影响，普通消费者缺乏计算依据。

**痛点二：品牌连锁咖啡的咖啡因数据不透明。** 库迪、瑞幸、星巴克等品牌极少公开每款产品的咖啡因含量数据，用户在享受便利的同时无法了解实际摄入量。

**痛点三：缺乏持续追踪手段。** 多数用户对每日咖啡因摄入仅有模糊印象，无法量化管理，更无法在接近安全阈值时获得提醒。

### 1.2 项目目标

构建一款运行于微信平台的咖啡因摄入记录小程序，以"简单记录、科学计算、可视化管理"为核心理念。用户能在30秒内完成一次咖啡因记录的录入，系统自动汇总当日摄入量并以图表展示趋势，当摄入量接近安全阈值时弹出提醒。

### 1.3 最终交付成果

CaffLog最终实现为一个功能完整的微信小程序，包含4个底部Tab页面与1个子页面：

| 页面 | 功能 |
|------|------|
| 记录页 | 自制咖啡计算 + 品牌饮品记录（8品牌，40+饮品） |
| 今日记录 | 环形进度条 + 当日记录列表 + 编辑/删除 |
| 历史记录 | 全部记录按日分组 + 本周柱状图 + 编辑/导出 |
| 我的 | 微信登录 + 累计统计 + 设置入口 |
| 设置 | 阈值配置 + 提醒开关 + 数据导出与清除 |

技术栈：微信小程序原生框架 + Canvas 2D自绘图表 + 云开发数据库与云函数 + 本地存储双写。

## 二、需求分析

### 2.1 功能性需求

经用例分析和用户场景梳理，本项目确定以下功能模块：

**P0 — 咖啡因摄入记录（两种模式）**

自制咖啡模式：用户选择咖啡豆品种（阿拉比卡/罗布斯塔/混合豆/低因咖啡）→ 输入咖啡豆重量（1-200g）→ 选择冲煮方式（手冲/意式浓缩/法压壶/冷萃/摩卡壶）→ 系统根据预设系数计算并显示估算值 → 用户确认保存。

品牌饮品模式：用户选择品牌（横向滚动8个品牌，含中英文名称）→ 选择饮品种类 → 选择饮品名称 → 选择杯型 → 系统匹配数据库中的咖啡因含量 → 用户一键保存。

**P0 — 每日摄入汇总**

首页展示当日咖啡因摄入总量（Canvas环形进度条）、剩余可用额度（按400mg上限计算）、当日摄入记录列表（按时间倒序，显示品牌/豆种及详细描述）。记录支持点击后编辑咖啡因量或删除。

**P1 — 历史记录与趋势**

历史记录页按日期分组展示全部记录，每日显示合计摄入量。提供本周（周一~周日）柱状图趋势，直观呈现每日摄入变化。支持记录的编辑、删除和按日批量删除。

**P1 — 用户设置**

支持开启/关闭预警提醒，自定义警告阈值（默认350mg，黄色提示）和超标阈值（默认400mg，红色警告）。提供数据导出（复制为格式化文本）、全部记录清除功能。

**P2 — 微信登录**

支持用户通过微信授权登录，获取头像和昵称展示在"我的"页面。登录信息持久化到本地存储。

### 2.2 非功能性需求

- **性能**：页面切换流畅，Canvas图表渲染延迟不超过300ms
- **兼容性**：支持微信客户端8.0及以上版本，覆盖iOS和Android
- **数据可靠性**：云数据库与本地存储双写策略，云端不可用时自动降级为纯本地模式
- **可用性**：核心记录操作不超过3步，界面简洁无技术术语
- **可维护性**：品牌饮品数据以独立JS模块管理，支持新增品牌无需修改核心代码

## 三、系统设计

### 3.1 整体架构

CaffLog采用经典的三层架构：

```
┌─────────────────────────────────────────┐
│         UI层 (WXML + WXSS)               │
│   记录页  今日记录  历史记录  我的  设置    │
├─────────────────────────────────────────┤
│         业务逻辑层 (Utils + Page.js)       │
│   calculator  brandService  recordService │
│   statsService  constants                │
├─────────────────────────────────────────┤
│         数据层 (Storage + Cloud)          │
│   本地Storage ←→ 云数据库 (双写)          │
│   云函数 caffeineService / dataInit       │
└─────────────────────────────────────────┘
```

**UI层**：基于微信小程序原生WXML+WXSS构建，图表使用Canvas 2D API自绘（不依赖第三方图表库）。自绘7个SVG/PNG图标用于Tab栏和功能标识。

**业务逻辑层**：5个核心工具模块：
- `calculator.js`：咖啡因计算引擎，实现 `咖啡因(mg) = 重量(g) × 品种系数(mg/g) × 冲煮萃取率(%)`，附带±20%波动范围
- `brandService.js`：品牌饮品数据服务，提供三级联动查询（品牌→品类→饮品→杯型）
- `recordService.js`：记录CRUD服务，云+本地双写
- `statsService.js`：统计聚合服务，按日/周/月维度计算趋势
- `constants.js`：全局常量（品种、冲煮方式、品牌列表、阈值默认值）

**数据层**：微信云开发数据库存储，配合本地`wx.Storage`实现双写策略。所有云调用带5秒超时保护，超时后自动熔断降级为纯本地模式，保证数据不丢失。

### 3.2 核心模块设计

#### 3.2.1 咖啡因计算引擎

自制咖啡因的估算基于以下线性模型：

```
咖啡因(mg) = 咖啡豆重量(g) × 品种咖啡因系数(mg/g) × 冲煮萃取率(%)
```

例如，15g阿拉比卡手冲 = 15 × 12mg/g × 88% ≈ 158mg（波动范围126-190mg，±20%）。

品种系数来源于学术文献和FDA标准：阿拉比卡12mg/g、罗布斯塔22mg/g、混合豆15mg/g、低因咖啡2mg/g。萃取率参考行业通用值：手冲88%、意式浓缩75%、法压壶90%、冷萃85%、摩卡壶80%。

#### 3.2.2 品牌饮品数据管理

品牌饮品数据以结构化JSON格式存储于`brandService.js`，每个饮品条目包含品牌ID、品类、饮品名称、杯型、毫升数、咖啡因含量等字段。支持8个品牌、40+款饮品。

三级联动选择通过函数式查询实现：`getCategories(brandId)` → `getDrinks(brandId, category)` → `getSizes(brandId, category, drinkName)` → `findCaffeine(...)`。预置数据同时提供给云函数`dataInit`用于数据库初始化。

#### 3.2.3 Canvas图表绘制

本项目选择使用Canvas 2D API自绘图表而非引入ECharts等第三方库，原因如下：
- 减少包体积，无需引入额外依赖
- 更高的绘制控制精度
- 适配小程序Canvas 2D接口

环形进度条：使用`arc()`绘制两个同心圆弧（背景环+进度环），根据占比动态切换颜色（绿→黄→红），中心叠加文字显示数值。

柱状图：用`fillRect()`绘制柱子，自动计算Y轴最大值（向上取整到百位），绘制400mg安全参考线（红色虚线），优化X轴标签显示避免拥挤。

折线图：`lineTo()`绘制平滑折线，`arc()`标记数据点，下方填充半透明渐变区域。

组件使用`observers`监听数据变化，Canvas未就绪时通过`_tryDraw()`重试机制（30ms×10次）确保绘制成功。

#### 3.2.4 数据存储双写策略

最大挑战是在云数据库集合尚未创建或网络不可用时保证应用正常运行。设计采用"本地优先、云端异步"策略：

- **写入**：立即写入本地`wx.Storage`并返回成功，异步写入云数据库（fire-and-forget，不阻塞）
- **读取**：优先从本地读取返回，同时异步从云端拉取并合并去重
- **超时保护**：所有云调用包装5秒超时，超时后设置`_cloudOk = false`熔断，后续调用跳过云端
- **恢复机制**：如果后续云调用成功，自动恢复`_cloudOk = true`

### 3.3 数据库设计

**集合1：intake_records（摄入记录）**

| 字段 | 类型 | 说明 |
|------|------|------|
| _id | String | 自动生成ID（`r_时间戳_随机串`） |
| record_type | String | `homemade` 或 `brand` |
| caffeine_mg | Number | 咖啡因毫克数 |
| recorded_at | String | ISO8601时间戳 |
| bean_name | String | 咖啡豆品种（自制） |
| bean_weight_g | Number | 豆重g（自制） |
| method_name | String | 冲煮方式（自制） |
| brand_name | String | 品牌名称（品牌） |
| drink_name | String | 饮品名称（品牌） |
| size_name | String | 杯型（品牌） |

**集合2-4**：`coffee_beans`（品种）、`brew_methods`（冲煮方式）、`brand_drinks`（品牌饮品数据）——预置只读数据，通过`dataInit`云函数初始化。

### 3.4 UI设计

**色彩系统**：主色咖啡棕`#6B4E3D`，背景奶白`#F5F0EB`，卡片纯白`#FFFFFF`，文字层级`#2C2416`/`#8B8178`，警告黄`#F0AD4E`，危险红`#E74C3C`。

**品牌选择**：横向滚动`<scroll-view>`，每个品牌标签上方中文名称、下方英文名称，居中对齐。选中状态切换为咖啡棕背景白字。

### 4.3 遇到的问题与解决方案

**问题一：品牌选择无法切换**
- 现象：点击品牌标签无反应
- 原因：品牌标签使用`bindtap`触发`onBrandChange`，但函数内按`<picker>`逻辑读取`e.detail.value`，取值为空
- 解决：将`onBrandChange`改为从`e.currentTarget.dataset.id`获取品牌ID

**问题二：记录保存失败**
- 现象：点击"确认记录"后提示"保存失败"
- 原因：`recordService`直接调用`wx.cloud.database()`，但云环境未配置且`db.serverDate()`兼容性问题
- 解决：改为本地存储方案，配合云数据库双写，云写入失败不影响本地使用

**问题三：Canvas图表不跟随数据更新**
- 现象：新增记录后切换到历史页，图表仍为空白或旧数据
- 原因1：使用`wx:if`控制图表显示，组件重建时`ready()`在Canvas布局完成前触发
- 原因2：云数据库查询超时阻塞了整个渲染流程
- 解决：移除`wx:if`使组件常驻，实现`_tryDraw(retryCount)`重试机制（30ms×10次），云调用加5秒超时+熔断

**问题四：编译错误**
- `@keyframes`CSS动画在WXSS中报"unexpected token"：移除`animation`和`@keyframes`，保留静态圆环
- Tab图标SVG格式报"仅支持png/jpg"：用Python Pillow生成PNG图标
- `wx:for-item="r"`与外层变量冲突：改为`wx:for-item="rec"`

**问题五：图表X轴标签过于密集**
- 现象：30日或本月柱状图X轴标签紧密重叠无法辨识
- 解决：柱状图模式在数据>7条时自动间隔显示日期标签；折线图模式不显示X轴标签

## 六、总结与展望

### 6.1 项目成果总结

CaffLog项目完整经历了从需求分析、系统设计、编码实现到测试优化的全流程开发。最终交付成果是一款功能完整的微信小程序：

- **核心能力**：自制咖啡因估算引擎（4种豆×5种冲煮）+ 品牌饮品快速记录（8品牌40+饮品）+ 每日仪表盘（环形进度条+记录列表）+ 历史趋势（本周柱状图）+ 阈值预警
- **技术亮点**：Canvas 2D自绘图表（零依赖）、云+本地双写存储策略（高可用）、5秒超时熔断机制（防卡死）、30ms重试的Canvas渲染保障
- **工程实践**：模块化工具层设计、组件化UI开发、观察者模式驱动数据更新、本地存储优先的降级策略

### 6.2 存在的不足与展望

**不足一：咖啡因计算模型的精度有限。** 当前线性系数模型忽略了研磨度、水温、萃取时间等变量，结果仅有±20%的参考意义。

**不足二：品牌饮品数据缺少官方来源。** 除星巴克部分产品外，多数品牌的咖啡因数据为行业均值估算，准确性有待验证。

**不足三：缺少社交和激励功能。** 纯工具型产品可能导致用户持续记录动力不足。

**后续优化方向：**
1. 引入多变量计算模型，支持用户校准个人化的换算系数
2. 探索品牌官方数据的自动化采集机制
3. 增加摄入打卡、周报推送等轻度激励功能
4. 考虑发布Web版（H5）和支付宝小程序等多端版本
