# CoffeeMan 咖啡因摄入记录微信小程序 — 需求文档

> 版本：v1.0 | 日期：2026-05-26 | 基于《CoffeeMan实践报告_修改版》整理

---

## 1. 项目概述

### 1.1 项目背景

中国咖啡市场在2025年已突破4000亿元规模，人均咖啡消费杯数从2019年的4.5杯/年增长至12杯/年以上。咖啡因作为一种中枢神经兴奋剂，适量摄入可提神醒脑，但过量摄入会导致心悸、焦虑、失眠，长期过量则与心血管疾病风险升高、钙流失等问题相关。FDA与EFSA均建议健康成年人单日咖啡因摄入量不超过400毫克，孕妇不超过200毫克。

当前健康管理类应用普遍聚焦于热量计算、运动记录和睡眠监测，缺乏专注于咖啡因摄入管理的工具。

### 1.2 核心痛点

| 痛点 | 描述 |
|------|------|
| 自制咖啡难以估算 | 手冲咖啡的咖啡因含量受豆种（阿拉比卡约1.2%、罗布斯塔约2.2%）、冲煮方式等多种因素影响，普通消费者缺乏计算依据 |
| 品牌饮品数据不透明 | 库迪、瑞幸、星巴克等品牌极少公开每款产品的咖啡因含量数据 |
| 缺乏持续追踪与预警 | 用户对每日摄入仅有模糊印象，无法量化管理，无法在接近安全阈值时获得提醒 |

### 1.3 产品定位

一款轻量级微信小程序，以"简单记录、科学计算、可视化管理"为核心理念。用户能在30秒内完成一次咖啡因记录的录入，系统自动汇总当日摄入量并以图表展示趋势，接近400mg安全阈值时通过订阅消息提醒。

### 1.4 目标用户

- **日常咖啡消费者**：希望了解并管理自身咖啡因摄入习惯的普通用户
- **健康管理人群**：因健康原因（如怀孕、心脏疾病）需要严格控制咖啡因摄入的人群
- **咖啡爱好者**：对不同咖啡豆品种和冲煮方式的咖啡因差异感兴趣的深度用户

## 2. 功能性需求

### 2.1 需求总览（按优先级排列）

| 优先级 | 模块 | 核心场景 |
|--------|------|----------|
| P0 | 自制咖啡因摄入记录 | 用户自制咖啡时，选择豆种、输入重量、选择冲煮方式，自动计算咖啡因含量 |
| P0 | 品牌饮品快速记录 | 用户在连锁咖啡店购买饮品后，通过品牌→饮品→杯型三级联动快速记录 |
| P1 | 每日摄入汇总与可视化 | 首页展示当日总量（环形进度条）、剩余额度、7日/30日趋势柱状图 |
| P1 | 历史记录管理 | 日历视图查询历史、支持编辑/删除/搜索 |
| P2 | 超标预警提醒 | 摄入超350mg黄色警告、超400mg红色警告，阈值可自定义 |

### 2.2 P0 — 自制咖啡因摄入记录

**场景描述**：用户在家用15克阿拉比卡咖啡豆制作了一杯手冲咖啡，想知道这杯咖啡大约含有多少毫克咖啡因。

**功能流程**：
1. 用户选择咖啡豆品种（下拉列表）
2. 用户输入咖啡豆使用重量（克）
3. 用户选择冲煮方式
4. 系统自动计算并显示估算咖啡因含量
5. 用户确认后保存至当日摄入列表

**计算模型**：

```
咖啡因(mg) = 咖啡豆重量(g) × 品种咖啡因系数(mg/g) × 冲煮萃取率(%)
```

**预设咖啡豆品种**：

| 品种 | 系数 | 数据来源 |
|------|------|----------|
| 阿拉比卡 (Arabica) | 12 mg/g | McCusker et al., 2003 |
| 罗布斯塔 (Robusta) | 22 mg/g | McCusker et al., 2003 |
| 阿拉比卡-罗布斯塔混合 | 15 mg/g | 行业均值 |
| 低因咖啡 (Decaf) | 2 mg/g | FDA标准 |

**预设冲煮方式**：

| 冲煮方式 | 萃取率 |
|----------|--------|
| 手冲 | 88% |
| 意式浓缩 | 75% |
| 法压壶 | 90% |
| 冷萃 | 85% |
| 摩卡壶 | 80% |

**计算示例**：15g阿拉比卡手冲 = 15 × 12 × 0.88 ≈ 158mg（显示范围：120-190mg）

### 2.3 P0 — 品牌饮品快速记录

**场景描述**：用户在瑞幸门店购买了一杯中杯美式咖啡，想快捷记录咖啡因含量。

**交互设计**：采用三级联动选择——品牌 → 饮品种类 → 杯型，与用户日常点单逻辑一致。

**覆盖品牌**：

| 品牌 | 杯型体系 |
|------|----------|
| 星巴克 (Starbucks) | Tall(354ml) / Grande(473ml) / Venti(591ml) |
| 瑞幸 (Luckin) | 中杯(355ml) / 大杯(473ml) |
| 库迪 (Cotti) | 中杯(355ml) / 大杯(473ml) / 超大杯(591ml) |

**数据说明**：
- 来源：品牌官方营养成分表、第三方检测报告、同类产品均值估算
- 每条数据标注来源和置信度（high/medium/low）
- 暂无官方数据的品类以行业均值填充，并明确标注"估算值"
- 精简版覆盖约20余款饮品

### 2.4 P1 — 每日摄入汇总与可视化

**首页仪表盘**：
- 当日咖啡因摄入总量（环形进度条，400mg上限）
- 剩余可用额度显示
- 当日摄入记录列表（按时间倒序）

**趋势图表**：
- 7日摄入趋势柱状图
- 30日摄入趋势柱状图
- 图表支持点击交互显示具体数值

> 选用柱状图而非折线图：柱状图在"按日对比绝对摄入量"场景下更直观，每根柱子的高度直接对应当日摄入总量，适合用户快速对比每日差异。

### 2.5 P1 — 历史记录管理

- 日历视图的历史记录查询，支持按日期浏览
- 每条记录支持编辑和删除操作
- 搜索过滤功能
- 分页加载（每次20条）

### 2.6 P2 — 超标预警提醒

**两级阈值**：

| 级别 | 阈值（默认） | 行为 |
|------|-------------|------|
| 警告 | ≥ 350mg | 黄色提示弹窗 |
| 超标 | ≥ 400mg | 红色警告弹窗 (wx.showModal) |

- 用户可在设置中自定义阈值
- 支持开启/关闭提醒
- 阈值存储于本地缓存（wx.setStorageSync），支持离线读取

## 3. 非功能性需求

### 3.1 性能

| 指标 | 要求 |
|------|------|
| 云数据库查询响应 | < 500ms |
| 小程序冷启动 | < 2s |
| 页面切换 | 流畅无明显卡顿 |
| 连续记录100条后 | 页面滚动流畅 |

### 3.2 兼容性

- 微信客户端 8.0 及以上版本
- 双端覆盖：iOS + Android
- 适配微信小程序标准分辨率（750rpx设计稿）

### 3.3 安全性

- 用户数据通过微信云开发存储，利用微信数据安全体系
- 不使用任何第三方数据统计SDK
- 遵循微信小程序隐私保护规范

### 3.4 可用性

- 核心功能（记录一次咖啡因摄入）操作步骤不超过3步
- 界面文案简洁、无技术术语
- 通过微信授权登录即可使用，无需额外注册

### 3.5 可维护性

- 品牌饮品数据以独立JS数据模块管理
- 支持通过云函数热更新数据，无需修改小程序代码
- 咖啡因计算参数支持配置化调整

## 4. 数据模型

### 4.1 数据集合定义

**集合1：coffee_beans（咖啡豆品种）**

| 字段 | 类型 | 说明 |
|------|------|------|
| _id | String | 自动生成 |
| name | String | 品种名称 |
| caffeine_per_gram | Number | 每克咖啡因 (mg/g) |
| origin | String | 产地 |
| is_custom | Boolean | 是否用户自定义 |

**集合2：brew_methods（冲煮方式）**

| 字段 | 类型 | 说明 |
|------|------|------|
| _id | String | 自动生成 |
| name | String | 冲煮方式名称 |
| extraction_rate | Number | 萃取率 (0-1) |
| description | String | 描述 |

**集合3：brand_drinks（品牌饮品）**

| 字段 | 类型 | 说明 |
|------|------|------|
| _id | String | 自动生成 |
| brand | String | 品牌 |
| category | String | 品类 |
| drink_name | String | 饮品名 |
| size_name | String | 杯型 |
| size_ml | Number | 毫升 |
| caffeine_mg | Number | 咖啡因 (mg) |
| data_source | String | 数据来源：official/estimated |
| confidence | String | 置信度：high/medium/low |

**集合4：intake_records（摄入记录）**

| 字段 | 类型 | 说明 |
|------|------|------|
| _id | String | 自动生成 |
| record_type | String | homemade / brand |
| caffeine_mg | Number | 咖啡因 (mg) |
| bean_id | String | 关联咖啡豆 (homemade时) |
| bean_weight_g | Number | 豆重 (homemade时) |
| method_id | String | 关联冲煮方式 (homemade时) |
| drink_id | String | 关联品牌饮品 (brand时) |
| recorded_at | String | ISO8601时间字符串 |
| notes | String | 备注 |

### 4.2 集合关系

```
coffee_beans (1) ──< intake_records (0..N)  [bean_id]
brew_methods (1) ──< intake_records (0..N)  [method_id]
brand_drinks (1) ──< intake_records (0..N)  [drink_id]
```

## 5. 系统架构

### 5.1 三层架构

```
┌─────────────────────────────────────────┐
│        UI Layer (WXML + TDesign)         │
│   首页  自制记录  品牌记录  历史  统计  设置  │
├─────────────────────────────────────────┤
│      Business Logic Layer (Utils)        │
│   calculator  brandData  recordService   │
│   stats                                 │
├─────────────────────────────────────────┤
│        Data Layer (wx.cloud DB)          │
│   coffee_beans  brew_methods             │
│   brand_drinks  intake_records           │
└─────────────────────────────────────────┘
```

### 5.2 技术选型

| 层级 | 技术方案 | 选型理由 |
|------|----------|----------|
| 前端框架 | 微信小程序原生 + TDesign组件库 | 即用即走、双端覆盖、零安装门槛、一套代码跨iOS/Android |
| 数据库 | 微信云开发数据库（NoSQL文档型） | 支持聚合查询（group by/sum/order by）、云函数处理复杂逻辑、免运维、免费额度充足 |
| 状态管理 | 原生 Page.setData() | 本项目状态需求简单（当日摄入列表+设置+品牌缓存），无需引入MobX等额外库 |
| 图表 | ECharts for WeChat | 同时支持环形图和柱状图、交互能力强、中文文档完善、MIT协议 |
| 数据初始化 | 云函数 | 首次部署时检测集合是否为空，若为空则批量写入品牌饮品数据 |

## 6. 页面结构

| 页面 | 路由 | 说明 |
|------|------|------|
| 首页仪表盘 | pages/index/index | 环形进度条 + 当日记录列表 + 趋势入口 |
| 自制咖啡记录 | pages/homemade/homemade | 选豆种 → 输重量 → 选冲煮 → 显示结果 |
| 品牌饮品记录 | pages/brand/brand | 选品牌 → 选饮品 → 选杯型 → 一键记录 |
| 历史记录 | pages/history/history | 日历视图 + 分页列表 + 编辑/删除 |
| 统计图表 | pages/stats/stats | 7日/30日柱状图 |
| 设置 | pages/settings/settings | 阈值配置 + 提醒开关 |

---

## 7. 数据同步策略

- 采用"每次 onShow 时重新拉取最新数据"策略
- 统计计算在云函数端用数据库聚合查询完成
- 首页仅加载当日和近7日摘要数据
- 历史页采用分页加载（每页20条）
- 删除记录后主动标记首页需刷新

## 8. 已知限制与后续方向

| 限制 | 后续优化方向 |
|------|-------------|
| 咖啡因计算模型为线性简化，忽略研磨度、水温、萃取时间等变量 | 引入多变量模型，支持用户校准个人化换算系数 |
| 品牌饮品数据手动收集，覆盖不全且可能过时 | 社区贡献机制或网络爬虫自动更新 |
| 纯工具型产品，缺乏持续记录动力 | 增加摄入打卡、周报推送、好友排行等轻度社交功能 |
| 依赖微信生态，非微信用户无法使用 | 抽离核心计算逻辑，扩展Web版(H5)和支付宝小程序等多端版本 |
